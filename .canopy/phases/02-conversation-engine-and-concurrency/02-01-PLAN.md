Implementation logic:

1. Initialize messages array with doctor's system prompt and opening message:
   ```
   messages = [
     { role: 'system', content: DOCTOR_SYSTEM_PROMPT },
     { role: 'user', content: DOCTOR_OPENING }
   ]
   ```

2. For each turn (1 to turnsPerConversation):
   a. Check `shouldStop?.()` — if true, break early
   b. Determine current role: odd turns are patient (API plays patient responding to doctor), even turns are doctor (API plays doctor responding to patient)
   c. Build the API messages: prepend the appropriate system prompt (patient or doctor), then include the full conversation history
   d. Call `config.makeRequest(apiMessages)`
   e. Append the response as an assistant message to the conversation history, then swap to user role for the next turn's perspective
   f. Emit `conversation:turn:complete` event with all metrics
   g. Log the turn via pino logger

3. After all turns (or on error):
   - Emit `conversation:complete`
   - Return ConversationResult

**Critical detail on message construction:**
The messages array should be structured so each API call sees the full conversation as if the API is one party. For the patient turn:
```
[patientSystemPrompt, ...conversationHistory]
```
Where conversationHistory alternates user (doctor says) / assistant (patient says).

For the doctor turn:
```
[doctorSystemPrompt, ...conversationHistory]
```
Where conversationHistory alternates user (patient says) / assistant (doctor says).

The trick: the conversation history is the SAME array but the roles are FLIPPED depending on who's "speaking" this turn. Each message in the shared history stores the actual role (doctor/patient) and the runner maps to user/assistant based on whose turn it is.

**Simpler approach (recommended):** Keep a single messages array that grows. On patient turns, prepend patient system prompt. On doctor turns, prepend doctor system prompt. The conversation history itself uses user/assistant roles consistently from one perspective — switch the system prompt to change who the API "is."

Actually simplest: maintain one messages array. Turn 1 (patient): system=patient, history starts with user="How are you feeling today?". API responds as assistant (patient). Turn 2 (doctor): system=doctor, history has user=patient_response (flipped from assistant). API responds as assistant (doctor). This requires building a fresh messages array each turn with role mapping.
